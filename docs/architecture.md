# Architecture

## Стиль системы

Модульный монолит с одним backend-образом и тремя режимами запуска:

- `api` — FastAPI и административный REST API;
- `worker` — singleton APScheduler для исходящих сообщений и напоминаний;
- `vk-listener` — Bots Long Poll listener.

PostgreSQL является системой учёта и надёжной очередью небольшого объёма. Redis,
Celery и отдельное объектное хранилище на первом этапе не используются.

## Основной поток

1. Listener получает событие неизвестной беседы и сохраняет `vk_chat`.
   Для каждого принятого сообщения или подходящей реакции обновляется время последней активности беседы.
2. Руководитель связывает беседу с учебной группой и классифицирует участников.
3. API одной транзакцией создаёт рассылку, цели, снимок получателей и исходящие
   задания.
4. Worker отправляет задания через VK API со стабильным `random_id` и уникальным
   `payload.broadcast_token`.
5. Listener связывает исходящее сообщение с заданием через payload, затем принимает reply или реакцию студента.
6. Ответ и метаданные вложений сохраняются транзакционно.
7. Listener скачивает изображения в persistent volume с проверкой источника, формата и размера; при запуске восстанавливает недостающие файлы.
8. Read-only analytics API агрегирует студентов, ответы и выполнение рассылок без изменения операционных данных.

## Инварианты

- Опубликованная рассылка неизменяема.
- Удаление рассылки каскадно очищает её локальные цели, задания, ответы и файлы; уже отправленные сообщения VK не изменяются.
- Состав получателей является снимком на момент создания.
- Новые участники имеют роль `unknown`; рассылка блокируется до классификации.
- На одну цель и этап существует не более одного исходящего задания.
- Worker повторно использует сохранённый `random_id`, поэтому повтор после сбоя идемпотентен на стороне VK.
- При старте worker один раз возвращает прерванные задания из `processing` в `pending`; задания со статусом `sent` повторно не захватываются.
- Окончательно неудачное задание можно вручную вернуть в очередь до дедлайна без сброса счётчика попыток и `random_id`.
- На одного получателя сохраняется последнее подходящее подтверждение; старые и повторно доставленные события его не перезаписывают.
- Сообщение принимается только как reply на сообщение текущего сообщества с известным `broadcast_token`; реакция — только на сохранённое исходящее сообщение рассылки.
- После дедлайна исходящие задания не выполняются.
- Поздние ответы сохраняются и помечаются отдельно.
- Все даты хранятся в UTC и отображаются в `Europe/Moscow`.

## Ограничение первого этапа

Файлы хранятся в Docker volume, поэтому установка рассчитана на один сервер.
Переход на несколько экземпляров потребует S3-совместимого хранилища.
